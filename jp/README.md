# Trading Automation Platform

> **現在、私自身が実運用している自動化システムです。**

このプロジェクトは、**複数の外部サービスを一つにつなぎ、外部から届いたデータを受け取って処理し、実際の作業を自動で実行し、その結果を確認し、サーバー上で止めることなく運用し続ける** ——そうしたシステムを、企画から開発・運用まで一人で構築した記録です。

具体的な題材は「取引の自動化（Trading Automation）」ですが、その中身は、外部 API 連携・Webhook 受信・複数サービスの統合・業務自動化・自動実行・通知/監視・クラウドサーバー運用といった、**あらゆる「連携型の自動化システム」に共通する土台** で成り立っています。

つまりこのリポジトリは、単なる取引ボットの紹介ではありません。「頭の中にあるアイデアを、実際に動いて回り続けるシステムとして形にできる人間かどうか」を確かめていただくための資料です。

---

## このシステムが一言で何をするのか（開発の知識がなくても分かる説明）

たとえば、こういう流れを想像してください。

1. **外部のサービス（TradingView）で「今、こういう状況になった」という合図が発生する。**
2. その合図が、**インターネット経由で自動的に私のシステムへ届く（Webhook）。**
3. システムが内容を判断し、**証券会社や取引所へ自動で注文を出す。**
4. 実行した結果を、**スマートフォンの Telegram に自動で通知する。**
5. すべての記録がサーバーに残り、**あとから正しく動いたかを確認できる。**

人が画面を見張って、手で操作して、結果を確認して……という一連の手作業を、**24時間止まらずに自動で回す** 仕組みです。

この「外部の合図を受けて → 自動で判断して → 実際の作業を実行して → 結果を通知・記録する」という骨組みは、取引に限らず、**予約管理・在庫連携・問い合わせ対応・データ収集・各種通知など、多くの業務自動化にそのまま応用できます。**

---

## こういう課題をお持ちの方へ

このシステムの構造は、次のようなご依頼と同じ技術で成り立っています。

- 「複数の外部サービス（決済・チャット・SNS・業務ツールなど）を連携させたい」
- 「Webhook や API で外部からデータを受け取り、自動で処理したい」
- 「手作業で繰り返している業務を自動で実行させたい」
- 「処理結果を Slack / Telegram / LINE などへ自動通知したい」
- 「クラウドサーバー（AWS）上で、止まらずに動き続けるシステムを運用したい」
- 「一度作って終わりではなく、運用しながら改善し続けられる仕組みがほしい」

取引システムをお求めでない方でも、**「自分がやりたい複数サービス連携型の自動化も、この人になら任せられそうだ」** と感じていただけるように、実際に運用中のシステムを通して技術力をお見せします。

# Overview

市場のシグナルが発生してから実際の注文が実行されるまでの全工程を、一つのプラットフォーム上で管理できるように構成しています。

現在このプラットフォームは、以下の機能を **一つの実行フロー** として統合管理しています。

- 市場シグナルの受信
- 注文の実行
- ブローカー（証券会社・取引所）連携
- 実行モニタリング
- サーバー管理
- デプロイ管理

個々の機能を単体で作り込むことよりも、**システム全体の実行フローを安定して回し続けること** を重視して設計しました。これは、実際に自分の資産を扱う環境で日々使っているからこその優先順位です。

---

# Key Features

## Real-Time Signal Processing（リアルタイムのシグナル処理）

TradingView の Webhook を通じて届く市場シグナルをリアルタイムで受信し、実行フローを開始します。**外部サービスから届くイベントを起点にシステムを動かす**、連携型自動化の中核部分です。

---

## Multi Broker Platform（複数ブローカーの統合）

一つのプラットフォーム上で、複数の取引環境をまとめて管理できるように構成しています。

**現在の対応環境**

- 韓国投資証券（한국투자증권）
- Toss 証券（토스증권）
- Upbit

ブローカーごとの実装は独立して管理し、プラットフォーム側では常に一貫した実行フローを維持するよう設計しています。**外部サービスごとに仕様が異なっても、共通の処理フローに吸収する** ——この構造は、複数の外部サービスを扱う統合システム全般に応用できます。

---

## Monitoring（モニタリング）

主要な実行結果を Telegram で確認できるように構成しています。

システムを起動しっぱなしにするだけでなく、**実行後も運用状態を継続的に把握できる監視環境** を用意しています。通知先は Telegram に限らず、Slack や LINE など他のチャットサービスへも同じ考え方で拡張できます。

---

## Deployment Process（デプロイ）

本番サーバーへの反映は、デプロイスクリプトをベースに管理しています。

アプリケーションをビルドしたあとにデプロイスクリプトを実行すると、本番反映に必要な作業が順番に自動で行われます。

デプロイ時には、次の作業を標準化された手順で進めます。

- 既存サービスの停止
- 最新アプリケーションのアップロード
- サービスの再起動
- 運用ログの接続

繰り返し発生する作業を減らし、**誰がやっても同じ結果になる一貫したデプロイ手順** を維持できるよう構成しています。運用と保守を見据えた設計です。
