# LINE AI Consultation Automation

> **日本の開発相談を、安心して続けられる仕組みへ。**
>
> AIを「回答するだけのチャットボット」としてではなく、
> **実際の開発相談を支えるシステム**として設計・運用しています。

<p align="center">

Java ・ Spring Boot ・ OpenAI API ・ LINE Messaging API ・ AWS

Production Architecture ・ Conversation Design ・ Reliability

</p>

---

# はじめに

AIチャットボットを作ること自体は、それほど難しくありません。

OpenAI APIとLINEを接続すれば、

ユーザーの質問に対して自然な文章を返すことはできます。

しかし、実際にサービスとして運用を始めると、
課題はまったく別のところに現れます。

例えば、

- 同じイベントが重複して送信される
- AIの応答が遅延する
- 外部APIが一時的に利用できなくなる
- 会話が数日にわたって続く
- ユーザーが複数のメッセージを連続して送信する
- AIが十分な根拠のない回答を生成する

こうした問題は、

AIの性能だけでは解決できません。

必要なのは、

**AIを安全に運用するためのバックエンド設計**です。

このプロジェクトは、その考え方をもとに構築しています。

---

# このプロジェクトについて

**LINE AI Consultation Automation** は、

日本向けのソフトウェア開発・業務自動化に関する相談を受け付けるためのシステムです。

目的は、

AIにすべてを任せることではありません。

自然な相談体験を維持しながら、

システム全体として一貫性と信頼性を保つことです。

そのため、

AIはシステム全体の一部として組み込み、

相談の流れや状態管理、

各種ビジネスルール、

外部サービスとの連携は、

バックエンドアプリケーションが担当します。

```text
LINE
    │
    ▼
Webhook
    │
    ▼
Backend Application
    │
    ├── Conversation Management
    ├── AI Interpretation
    ├── Business Rules
    ├── Validation
    └── Response Delivery
```

この構成により、

AIの出力品質だけに依存しない、

安定した相談システムを実現しています。

---

# なぜ、このような設計にしたのか

多くのAIアプリケーションは、

「AIが回答を返すこと」

をゴールにしています。

しかし、

実際の開発相談では、

回答を返すことよりも、

**相談を最後まで自然に進められること**

の方が重要です。

例えば、

ユーザーが料金について質問した場合でも、

要件が決まっていなければ、

正確な見積もりはできません。

また、

納期だけを聞かれても、

必要な機能が分からなければ判断はできません。

つまり、

相談とは、

質問に一つずつ答えることではなく、

必要な情報を整理しながら、

要件を少しずつ明確にしていくプロセスです。

本システムでは、

その相談の流れそのものを設計対象としています。

# 設計思想

このプロジェクトには、一つの基本方針があります。

> **AIを信頼するのではなく、AIを活用できる仕組みを設計すること。**

AIは自然言語を理解し、

相談内容を整理することが得意です。

一方で、

システム全体の整合性や、

業務上の判断、

データの管理、

外部サービスとの連携については、

バックエンドが責任を持つべきだと考えています。

そのため、

本システムでは、

AIとアプリケーションの責務を明確に分離しています。

AIは「考える」役割。

アプリケーションは「判断し、実行する」役割。

それぞれが適切な責任範囲を持つことで、

運用時の信頼性を高めています。

---

# このプロジェクトで大切にしていること

設計や実装を行う際は、

常に次の考え方を基準にしています。

- AIをシステムの一部として利用すること
- 会話ではなく「相談」を設計すること
- ビジネスルールはバックエンドで管理すること
- 外部サービスの障害を前提に設計すること
- 保守しやすく、改善し続けられる構成にすること

AIの性能だけに依存しない。

それが、このプロジェクト全体を通した設計思想です。

# アーキテクチャ設計

このシステムでは、

AIをシステムの中心には置いていません。

中心にあるのは、

**バックエンドアプリケーション**です。

AIは自然言語を理解するためのコンポーネントとして組み込み、

最終的な判断やシステム制御は常にアプリケーションが担当します。

```text
                 User
                  │
                  ▼
         LINE Messaging API
                  │
                  ▼
      Backend Application
                  │
        ┌──────────────┐
        │              │
        ▼              ▼
 Conversation      AI Analysis
   Management
        │              │
        └──────┬───────┘
               ▼
      Business Decision
               │
               ▼
      Response Delivery
```

この責務分離によって、

AIの出力だけに依存しない、

一貫性のある相談システムを実現しています。

---

# AIは「判断者」ではない

本システムでは、

AIを意思決定エンジンとして扱っていません。

AIの役割は、

- 相談内容を理解する
- ユーザーの意図を整理する
- 会話の流れを補助する

ことです。

一方で、

- ビジネスルール
- 会話状態の管理
- 外部サービスとの連携
- メッセージ送信
- システム状態の更新

については、

すべてバックエンドが担当します。

つまり、

AIは「提案」を行います。

最終的に採用するかどうかを判断するのは、

アプリケーションです。

この責任分離は、

システム全体の信頼性を維持するための重要な設計方針です。

---

# 情報には役割がある

AIが誤った回答を生成する原因の一つは、

異なる性質の情報を同じように扱ってしまうことです。

そのため本システムでは、

情報を目的ごとに整理しています。

```text
Business Policy
        │
        ▼
Verified Knowledge
        │
        ▼
Conversation Guidance
        │
        ▼
Current Consultation
```

それぞれが異なる責任を持つことで、

事実と会話表現が混在しない構成を目指しています。

重要なのは、

AIへ大量の情報を渡すことではありません。

**必要な情報を、必要なタイミングで提供すること**です。

---

# コンテキスト設計

相談内容は毎回異なります。

そのため、

すべての情報を毎回AIへ渡すことはしていません。

現在の相談内容や会話の流れに応じて、

必要なコンテキストのみを組み合わせます。

```text
Conversation Context
          │
          ▼
Relevant Knowledge
          │
          ▼
Business Policies
          │
          ▼
Prompt Composition
```

これにより、

不要な情報によるノイズを減らし、

相談内容に集中した応答を実現しています。

また、

コンテキスト構築の詳細なロジックは公開していません。

READMEでは、

実装方法ではなく、

設計思想を共有することを重視しています。

# 事実性を最優先にする

自然な日本語を話すことと、

正しい情報を伝えることは別の問題です。

本システムでは、

流暢さよりも、

**事実性**を優先しています。

十分な根拠がない内容については、

AIが推測によって断定しないよう設計しています。

例えば、

- 開発費用
- 納期
- 対応可能な機能
- 実績
- 外部サービスの仕様

などについては、

必要な情報が揃うまで結論を出しません。

相談を前に進めるために、

まず必要な情報を整理することを重視しています。

# 一問一答ではなく「相談」を設計する

このシステムは、

チャットボットではありません。

開発相談を支援するシステムです。

そのため、

ユーザーから質問が届くたびに回答を返すのではなく、

相談全体の流れを考慮します。

```text
Current Situation
        │
        ▼
Business Goal
        │
        ▼
Requirements
        │
        ▼
Missing Information
        │
        ▼
Next Conversation
```

相談では、

必要な情報は一度に揃いません。

会話を重ねながら、

少しずつ要件を整理していきます。

その流れを自然に支援することが、

このシステムの目的です。

---

# 会話状態を管理する

相談は一回のメッセージでは終わりません。

予算、

納期、

必要な機能、

現在の運用状況など、

情報は複数回のやり取りを通して集まります。

そのため、

本システムでは、

個々のメッセージではなく、

相談全体を一つの状態として管理しています。

これにより、

すでに確認した内容を繰り返し質問することなく、

前後の文脈を考慮した自然な相談フローを実現しています。

# 信頼性を前提とした設計

AIを利用したサービスでは、

「正常に動作すること」だけを前提に設計することはできません。

実際の運用では、

外部サービスの障害、

ネットワーク遅延、

AIの応答時間、

予期しない入力など、

さまざまな要因によってシステムの挙動は変化します。

そのため本システムでは、

**障害は例外ではなく、設計対象の一部**

であると考えています。

---

# Production First

このプロジェクトは、

デモやサンプルコードとして設計したものではありません。

実際の運用を前提とし、

長期間安定して利用できることを目的としています。

そのため、

AIだけではなく、

バックエンド、

外部API、

メッセージングプラットフォーム、

ネットワーク、

それぞれの責任範囲を明確に分離しています。

一つのコンポーネントで問題が発生しても、

システム全体へ影響が広がらない構成を目指しています。

---

# Reliable Conversation Processing

開発相談では、

ユーザーが複数のメッセージを短時間で送信することは珍しくありません。

例えば、

```text
「LINE Botを作りたいです。」

「予約機能も必要です。」

「予算は300万円くらいです。」
```

これらは独立した質問ではなく、

一つの相談です。

そのため、

本システムでは、

個々のメッセージではなく、

**相談全体の流れ**

を基準に処理しています。

会話の前後関係を維持しながら、

必要な情報を整理し、

自然な相談体験を提供できるよう設計しています。

---

# Asynchronous Architecture

外部サービスとの通信には、

常に一定の処理時間が発生します。

もし、

すべての処理を同期的に実行すると、

応答遅延やタイムアウトが発生しやすくなります。

そのため、

受信処理と内部処理を適切に分離し、

利用者への応答性を維持できる構成を採用しています。

```text
Request

     │

     ▼

Accept

     │

     ▼

Internal Processing

     │

     ▼

Response
```

重要なのは、

AIが速いことではありません。

**システム全体が安定して応答し続けられること**です。

---

# Defensive System Design

AIは非常に優秀ですが、

常に期待どおりの出力を返すとは限りません。

そのため、

AIの生成結果をそのまま利用することはありません。

アプリケーション側で内容を確認し、

必要な条件を満たした場合のみ、

後続処理へ進みます。

```text
AI

     │

     ▼

Validation

     │

     ▼

Business Rules

     │

     ▼

Application Decision
```

AIは、

システムへ提案を行います。

実際の判断を行うのは、

バックエンドアプリケーションです。

---

# Resilience

サービス運用では、

問題が起きないことよりも、

問題が起きた後にどう振る舞うかが重要です。

本システムでは、

外部サービスの一時的な障害や、

AIの応答遅延が発生した場合でも、

利用者にできる限り影響を与えない構成を目指しています。

完全に障害をなくすことはできません。

しかし、

障害が発生しても相談を継続できることは設計できます。

それが、

本プロジェクトで重視している考え方です。

---

# Observability

実際の運用では、

システムが動いていることだけでは十分ではありません。

「何が起きているか」を把握できることも重要です。

そのため、

各コンポーネントの処理状況を追跡できるように設計し、

問題発生時には原因を迅速に特定できる構成を採用しています。

運用とは、

障害をゼロにすることではなく、

障害に気付き、

改善し続けることだと考えています。

---

# Architecture for Long-Term Maintenance

システムは、

完成した瞬間から保守が始まります。

そのため、

将来的な機能追加や改善を見据え、

各コンポーネントを独立した責任で設計しています。

```text
LINE Platform

       │

       ▼

Backend

       │

 ┌─────┴─────┐

 ▼           ▼

Conversation   AI

       │

       ▼

Validation

       │

       ▼

       │

       ▼
```

責務を明確に分離することで、

機能追加や仕様変更が発生した場合でも、

影響範囲を最小限に抑えられる構成を目指しています。
